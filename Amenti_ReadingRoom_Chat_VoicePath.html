The Reading-Room Chat Voice Path

The most concrete latency finding of the session: in the Page2 reading room, a figure’s chat reply is voiced by an unchunked, non-streaming call — the worst first-sound path in the system, in the one place a person is actively waiting. This note records the finding and the fix now applied: an auto-switch by reply length, routing long replies through the streaming pipeline.

The three voice paths on Page2

Tracing the reading room turned up not two pipelines but three, each with different latency behaviour:

AmentiAudio
Long-form read-aloud. Chunks at ~700 chars, LOOKAHEAD=3, streams gaplessly. First chunk plays almost immediately. Built to fix the 524 timeout on essays. Streams ✓
Atlantica
Broadcasts. Also routes through AmentiAudio — same 700-char streaming pipeline. Streams ✓
Sovereign.Voice
The tablet chat reply. A single fetch(/speak, whole text) then play(). No chunking, no streaming, no lookahead. The listener hears nothing until the entire reply has rendered. Batches ✗

The finding, exactly located

When a figure replies in the reading-room chat, it is voiced here — appendAssistant(), the one and only chat voicing route (verified: Sovereign.Voice.speak is called in exactly one place):

// Page2.html, line 9286 — inside appendAssistant(text, who)
if (window.AMENTI_VOICE_ON) {
  Sovereign.Voice.speak(text, Sovereign.State.activeRecord);
}

And Sovereign.Voice.speak is the unchunked single-shot: await fetch(WORKER, { whole text }) → play(). Its own comment marks it as early scaffolding — “Phase 1: one Gemini voice; per-figure voices come in Phase 2.” It was a first hookup, never the streaming pipeline.

The consequence, against the measured variance: a cache-miss render ran 5–16 seconds tonight. Because this path waits for the whole reply before any sound, a reading-room chat answer can sit in silence for ten seconds or more — while the read-aloud pipeline inches away is already speaking on its first chunk. The slowest first-sound in the system sits in the one place a person is actively waiting for a reply.

Why it matters here specifically

The reading room is a sequence: it opens as long-form recitation, then devolves into casual chat at the end. The recitation half is voiced by the streaming AmentiAudio and feels immediate. Then conversation begins — and voicing silently drops to the non-streaming Sovereign.Voice path. The experience degrades at exactly the moment it should feel most responsive: a back-and-forth exchange.

The fix — implemented: auto-switch by length

The clean solution is not to pick a pipeline by mode (which drifts as the reading room devolves into chat) but by the length of the reply text itself — a fact already in hand the moment before voicing. Short replies keep the simple single-shot path (fast enough whole); long replies route to the streaming pipeline. The switch lives inside Sovereign.Voice.speak, so the single call site at line 9286 is untouched and every caller benefits.

IMPLEMENTED in Sovereign.Voice.speak

STREAM_THRESHOLD: 700,   // tune by listening; aligned with AmentiAudio TARGET_CHARS
async speak(text, rec) {
  try {
    var comp = this.composeFor(rec);
    // LONG -> streaming pipeline, SAME style+voice so cache keys stay aligned
    if (text && text.length > this.STREAM_THRESHOLD
        && window.AmentiAudio && window.AmentiAudio.play) {
      if (this.current) { try { this.current.pause(); } catch (e) {} }
      window.AmentiAudio.play(text, { style: comp.style, voice: comp.voice });
      return;
    }
    // SHORT -> the original single-shot path (unchanged below)
    const res = await fetch(this.WORKER, { ... });
    ...
Why this is the right design: it keys off the text, not a mode label — and the text is the ground truth. It reuses both pipelines already in the file; no new one is built. It preserves per-figure voice by passing composeFor’s style and voice through to AmentiAudio. And crucially, because a long reply that fits in one 700-char chunk sends the whole text with the same style+voice, its cache key is identical to what the short path would produce — so the boundary introduces no cache fragmentation.
The honest limit, unchanged: a genuinely short reply (under 700 chars) still takes the single-shot path and does not stream — but it does not need to, because a short reply renders fast whole. The win lands where it matters: long replies now start speaking on their first chunk instead of after the entire render. If short-reply latency ever bites, lowering the threshold (and accepting smaller chunks) is the follow-on tuning — a change to chunk behaviour the Speech Doctrine says to make deliberately, not casually.

The two pages are in different states

After the Page2 fix, the reading-room voice question was traced on Page1 as well — and the two pages turn out to have taken different roads. Page1 is further along: it has already done the deeper unification that Page2 has not.

Page1 — one unified streaming pipeline

Page1 removed its separate 700-char read-aloud pipeline entirely. Its own code says so plainly: “The old window.AmentiAudio pipeline (700/1100-char chunks) has been removed. All long-form readers on this page now call the shared reading-room engine via Amenti.throttle.speak. One chunker, one strip, one style composition across every surface.” Every voice call site on Page1 — the recitation, the questions after, Atlantica, the daily reader — routes through the same Amenti.throttle.speak, which chunks at 320 and streams. There is no pipeline to switch between, because there is only one. So in a Page1 reading room, a long reading and the chat that follows it are voiced the same way, both streaming, both fast to first sound.

Page2 — two pipelines, now bridged by the auto-switch

Page2 still runs two separate pipelines: AmentiAudio (700-char, streaming) for long-form, and, until tonight, the unchunked single-shot Sovereign.Voice for chat. The auto-switch bridges them — long replies now route to the streaming pipeline — but they remain two distinct engines with two chunk sizes and two cache namespaces beneath them.

PAGE 1
Reading: throttle (320, streams). Chat: throttle (320, streams). Atlantica: throttle. One unified pipeline, one shared cache. No autodetector needed — nothing to switch between.
PAGE 2
Reading: AmentiAudio (700, streams). Chat: Sovereign.Voice (now auto-switches to AmentiAudio for long replies). Two pipelines, bridged. Short chat replies still take the single-shot path; the two engines keep separate caches.
The plain read: Page1 solved this the thorough way (unify everything onto one streaming chunker), and Page2 solved it the light way (route long replies to the streaming pipeline it already had). Both now stream long readings and the chat that follows — but Page1 also gets a single shared cache from the unification, which Page2 does not yet have.

Proposed future changes

Nothing below is urgent, and the system has not yet been tested by live users — so these are directions, to be weighed and measured, not committed to now.

1 · Bring Page2 up to Page1’s design

The real end-state is to unify Page2 onto the throttle as well — retiring both Sovereign.Voice and AmentiAudio, routing every Page2 surface through Amenti.throttle.speak, exactly as Page1 does. That would give Page2 one streaming pipeline and the single shared cache it currently lacks, so a fragment paid for on either page is free on both. This is the thorough fix; the auto-switch was the bridge to it. Cost: it is a real refactor, and per the Speech Doctrine the chunk boundary is load-bearing — changing Page2’s chunking re-keys its cache once. Do it deliberately, with a re-render expected.

2 · Tune the chat chunk size for fast first sound

Both pages currently favour throughput-sized chunks for reading (Page1 at 320, Page2’s AmentiAudio at 700). For casual chat, a smaller first chunk renders faster and starts sooner. A future refinement: let the chunker use a smaller first-chunk size in chat mode, so the opening words of a reply land almost instantly, then widen for the tail. This is a chunk-boundary change — load-bearing, measure before touching.

3 · Pre-warm common opening measures

Replies and readings often begin similarly. Pre-rendering a small bank of common opening measures would make the first chunk of many replies a cache hit — served in well under a second while the novel remainder renders behind it. Cheapest of the three, exploits the streaming already in place, touches no chunk boundary.

4 · The bigger lever, unchanged: stream the LLM handoff

All of the above concern the voice half. The larger latency win named in the Latency Rundown still stands: today the full LLM reply is awaited before voicing begins. Streaming the LLM text into the voice engine sentence by sentence would overlap the two halves — the deepest win, and the most entangled with the conversation state machine. It remains gated on the open question of whether the completion API can stream.

The order that respects the doctrine: pre-warm (cheap, safe) → unify Page2 onto the throttle (thorough, a deliberate re-key) → tune chat chunk size (load-bearing, measure first) → stream the LLM handoff (biggest, most entangled). And after every one: listen. Being loaded is not being used.
