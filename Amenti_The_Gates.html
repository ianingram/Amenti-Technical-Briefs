TheGATES

Six Hearts · Every Question Pays Once · The Gate Opens

In the Book of the Dead the weighing is not the end. Afterwards the dead pass through a series of gates, and each opens only to one who knows the name. Not judgement — knowledge, tested at a threshold. That is what this is.

EVERY QUESTION PAYS ONCE — WHENEVER YOU FIRST GET IT RIGHT.
THE WEIGHING HAPPENS ONCE. THE GATE MAY BE RETURNED TO UNTIL IT OPENS.

What this changes

Today a quiz is a test: one attempt, a score, an end. The proposal is that it becomes a lock. Six hearts, and the next set of questions on that figure opens. A seeker does not pass or fail — they either know it yet or they do not.

The move that makes it work is separating two things that were tangled together. The weighing is once. The questions are practice until mastered.

whyyou do not re-weigh a heart. The cases are already argued and endorsed; re-filing them would corrupt the docket.knowing a thing later is still knowing it, and the work is the same work. A gate that never opens is not a gate, it is a wall.

The payout, revised

An earlier draft held that the first attempt set the tier and a retry paid nothing — progress without money. That was a rule invented to prevent a farm.

The rule is unnecessary, because the ledger already prevents it. Every correct answer writes quiz:topic:qid — a reference that is per question, not per attempt. A question can therefore only ever pay once, whenever it is first answered correctly. A retry pays only for hearts not yet held. There is nothing to enforce; the reference enforces it.

So the simpler design is also the safer one: each question pays when you get it right. Go back a week later, answer the one you missed, and you are paid for that one. The four you already hold pay nothing, because they already did.

ONE THING MUST HOLD FOR THIS TO STAY SAFE:
A WRONG ANSWER MUST NOT REVEAL THE CORRECT ONE.

If it did, a seeker could answer deliberately wrong to scout the answer, then retry and take the full timed tier for speed. Without disclosure a retry is exactly what it appears to be — I went away and learned it. That is the whole guard, and it costs nothing to keep.

The two bonuses

Two separate achievements, and they should not be confused. Each needs its own reference so each pays once.

ReferenceEarned byWhat it rewards
quiz:{topic}:completeall six hearts held, however long it tookpersistence — you came back until you knew it
quiz:{topic}:flawlessall six in a single sittingmastery — you knew it walking in

The existing MASTERY_BONUS = 50 becomes the second. The completion bonus should sit meaningfully below it: persistence is real and worth paying for, but it is not the same achievement, and the gap between the two numbers is what makes the flawless run worth attempting.

Both are references, not flags — so a seeker who eventually completes a quiz gets the completion bonus once, and a seeker who sweeps it first time gets both: the flawless bonus AND the completion, since completing it is what they did. That is correct. The flawless run is strictly better, and the arithmetic says so without a special case.

The hearts already exist

Every correct answer already writes a row: reason: 'quiz_pass', reference: 'quiz:manlius-capitol:geese'. That reference is unique per seeker, per question. Six correct answers are six rows.

So which hearts a seeker holds is already recorded. Nothing new is stored. What is missing is one endpoint that reads it back.

Figure 2 · the data is there. One read, and the query that must precede all of it.

RUN THE QUERY BEFORE ANYTHING ELSE.
IF RETAKES ALREADY PAY TWICE, THAT IS A LEAK IN THE LEDGER TODAY — WITH OR WITHOUT GATES.

The build

Gate IKnow What Is Already Truefirst

One query, before a line is written.

select count(*)                as rows,
       count(distinct reference) as refs
from emerald_ledger
where reason = 'quiz_pass';

Equal — the RPC refuses duplicates. A retry cannot pay again, and the whole design is safe by construction.

Unequal — the same question has already paid more than once. That is a leak that exists today, independent of this spec, and it is fixed before anything is built on top of it.

Done when
You know which of those two sentences is true.
Gate IIThe Progress Endpointone read

GET /quiz/progress on the mint, authenticated as the seeker. It reads their quiz_pass references, groups by topic, and returns which question ids are held.

{ "manlius-capitol": { "held": ["geese","surname","rock"], "of": 6, "mastered": false },
  "caesar-rubicon":  { "held": ["river","legion","why-war","die-cast"], "of": 4, "mastered": true } }

Note what it does not return: no answers, no keys, no correctness for questions not yet held. A question the seeker has not got right is simply absent — which is the same information a hollow heart carries, and nothing more.

Done when
A seeker's own hearts can be read back, and a stranger's cannot.
Gate IIIThe Retrypays what is not yet held

Re-entering a quiz already taken offers only the questions not yet held. The passage is there to read. The two cases are not asked again — they are already on the docket.

doesopen the heart, and pay for that question, at the tier earned on this attempt
doespay quiz:{topic}:complete when the sixth heart lands
does NOTpay again for a heart already held — the reference refuses it
does NOTpay flawless, which only a single unbroken sitting earns
does NOTre-file the cases, re-run the tally, or alter the verdict
Note what does not need building: no attempt counter, no "first pass" flag, no per-topic state table. The ledger already carries every fact this needs, because the reference was written per question rather than per session. That decision was made long before anyone thought about gates, and it is the reason this is a small build.
Done when
A retry pays for a heart newly opened and cannot move the balance for one already held.
Gate IVThe Card Shows Itthe point of all of it

Six hearts on the card — filled for held, hollow for not. The score beside each. And beneath, the state of the next gate.

This is what makes the arena a collection rather than a menu: a seeker scanning the roster sees at a glance where they are incomplete, and the hollow heart is an invitation rather than a scold.

Decide before building
  1. What does a signed-out visitor see? If hearts appear only after sign-in, the arena looks emptier to a stranger than to you — and the stranger is the one being persuaded. A greyed set with "sign in to begin" is likely better than nothing.
  2. Is the next set gated or signposted? "Six hearts to open the next" is a real unlock and a real risk: a seeker stuck on one question is stuck entirely. Signposting is softer and cannot trap anyone.
  3. Does the sixth heart look different? The moment a gate opens is the strongest beat in the system. It should not look like the fifth.
Done when
A returning seeker can see, without opening anything, which gates they have passed and which are one answer from opening.

What this supersedes

THE WEIGHING, Movement IV — "the punched card" — is struck. It proposed three hearts meaning three quizzes completed. This is six hearts meaning six questions mastered: a different unit, a smaller grain, and a stronger idea. The Weighing keeps Movements I to III, which are about what a seeker sees. This document governs what a seeker can do.

A note on the other side of the ledger

This adds faucets: every question pays, and two bonuses sit on top. The spending side already exists in part — VOTE_COST = 5 per endorsement is a genuine spend, live today, and it is the only one.

Two things are worth establishing before the shop is designed, and neither blocks this build:

1Is the endorsement a burn or a transfer? If the 5 ET funds the weekly pool it is redistributed and the supply never falls. If it vanishes, the economy genuinely contracts somewhere.
2Where does POOL_SIZE = 500 come from? Minted fresh each week makes the pool a faucet of its own — which may be correct, but should be a decision rather than an accident.
The relic shop, trade fees and endorsement burns were all designed in THE EXCHANGE and none are built. Rewards first is the right order — you cannot price a shop before you know what people hold. But the answers to those two questions decide what the shop has to do, so they are worth knowing early.
